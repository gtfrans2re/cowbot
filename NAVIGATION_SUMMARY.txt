--------------------------------------------------------------------------------
HOW IT WORKS
--------------------------------------------------------------------------------

The system uses a three-layer architecture:

1. ROBOT INTERFACE NODE (robot_interface)
   - Subscribes to: /scan, /odom, /imu
   - Publishes to: /cmd_vel
   - Exposes parameters for easy scripting access
   - Processes LiDAR data into sectored ranges

2. CONTROL LAYER (Python or Bash)
   - Reads sensor data via parameters
   - Implements obstacle avoidance algorithm
   - Sets velocity commands via parameters

3. HARDWARE LAYER (your existing system)
   - Motor driver executes velocity commands
   - LiDAR provides scan data
   - Odometry and IMU provide position/orientation

--------------------------------------------------------------------------------
OBSTACLE AVOIDANCE ALGORITHM
--------------------------------------------------------------------------------

1. Drive forward while front is clear (> threshold)
2. When obstacle detected, stop and evaluate all directions
3. Choose direction with maximum clearance
4. Turn until front clears
5. Resume forward motion

Tunable parameters:
- forward_speed: 0.2 m/s (default)
- turn_speed: 0.5 rad/s (default)
- threshold: 0.3-0.4 m (default)

--------------------------------------------------------------------------------
TROUBLESHOOTING
--------------------------------------------------------------------------------

If robot doesn't move:
1. Verify robot_interface is running
2. Check: ros2 topic echo /cmd_vel
3. Check: ros2 param list | grep robot_interface

If no obstacle detection:
1. Verify LiDAR: ros2 topic echo /scan
2. Check parameters: ros2 param get /robot_interface scan_front_ray_range

If robot behaves strangely:
1. Adjust threshold in control scripts
2. Verify LiDAR mounting and field of view
3. Check for sensor noise or interference
